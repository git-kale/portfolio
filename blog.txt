<h2>Introduction</h2>
<p>Microservices architecture has become the standard way to build large-scale distributed systems. <h2>The Microservices Paradigm</h2>
<p>Traditional monolithic applications become increasingly difficult to maintain and scale as they grow. Microservices architecture solves this by:</p>
<ul>
<li>Breaking down applications into small, focused services</li>
<li>Enabling independent deployment and scaling</li>
<li>Allowing teams to work independently</li>
<li>Improving fault isolation</li>
</ul>
<h2>Key Architectural Patterns</h2>
<h3>Service Discovery</h3>
<p>Service discovery is the process of automatically locating services in a dynamic environment. For Golang microservices, we typically use:</p>
<pre><code class="language-go">// Example with Consul
type ServiceRegistry interface {
    Register(service *Service) error
    Deregister(serviceID string) error
    Discover(serviceName string) ([]Service, error)
}
</code></pre>
<h3>Load Balancing</h3>
<p>Proper load balancing ensures requests are distributed efficiently:</p>
<ul>
<li>Round-robin distribution</li>
<li>Least connections</li>
<li>Weighted round-robin</li>
<li>IP hash-based</li>
</ul>
<h3>Fault Tolerance</h3>
<p>Building resilient systems requires:</p>
<ul>
<li>Timeouts and retries</li>
<li>Circuit breaker pattern</li>
<li>Health checking</li>
<li>Graceful degradation</li>
</ul>
<h2>Performance Considerations</h2>
<p>Golang offers:</p>
<ul>
<li>Efficient goroutines for concurrent operations</li>
<li>Fast binary compilation</li>
<li>Built-in HTTP support</li>
<li>Low memory footprint</li>
</ul>
<h2>Best Practices</h2>
<ol>
<li><strong>Keep services small and focused</strong> - Single responsibility principle</li>
<li><strong>Implement proper logging</strong> - Structured logging for debugging</li>
<li><strong>Use message queues</strong> - For asynchronous communication</li>
<li><strong>Monitor and observe</strong> - Distributed tracing and metrics</li>
<li><strong>Document APIs</strong> - Clear API contracts</li>
</ol>
<h2>Conclusion</h2>
<p>Building scalable microservices with Golang requires understanding both the architectural patterns and the language&#39;s capabilities. By following these practices and patterns, you can build systems that are both performant and maintainable.</p>
19:Te22,<h1>System Design: Caching Strategies at Scale</h1>
<h2>Why Caching Matters</h2>
<p>Caching is one of the most powerful tools for improving system performance. A well-designed caching strategy can reduce database load by 60-80% and improve latency significantly.</p>
<h2>Common Caching Patterns</h2>
<h3>Write-Through Cache</h3>
<p>Data is written to both cache and database simultaneously, ensuring consistency.</p>
<pre><code>Client -&gt; Cache -&gt; Database
         wait    (sync write)
</code></pre>
<p><strong>Pros</strong>: Guarantees data consistency
<strong>Cons</strong>: High write latency</p>
<h3>Write-Behind Cache</h3>
<p>Data is written to cache first, then asynchronously synchronized to the database.</p>
<pre><code>Client -&gt; Cache (async) -&gt; Database
         (return immediately)
</code></pre>
<p><strong>Pros</strong>: Low write latency, high throughput
<strong>Cons</strong>: Risk of data loss if cache fails</p>
<h3>Write-Around Cache</h3>
<p>Data bypasses the cache on write and goes directly to the database.</p>
<pre><code>Client -&gt; Database
Client -&gt; Cache (on read)
</code></pre>
<p><strong>Pros</strong>: Prevents cache pollution
<strong>Cons</strong>: Cache miss on first read</p>
<h2>Eviction Policies</h2>
<h3>LRU (Least Recently Used)</h3>
<p>Removes the least recently accessed item when capacity is reached.</p>
<p><strong>Best for</strong>: General-purpose caching with temporal locality</p>
<h3>LFU (Least Frequently Used)</h3>
<p>Removes the least frequently accessed item.</p>
<p><strong>Best for</strong>: Workloads with clear popularity patterns</p>
<h3>FIFO (First In First Out)</h3>
<p>Removes the oldest item first, regardless of access patterns.</p>
<p><strong>Best for</strong>: Simple TTL-based caching</p>
<h2>Cache Invalidation</h2>
<p>The hardest problem in computer science! Key strategies:</p>
<ol>
<li><strong>Time-based</strong>: Expire entries after a TTL</li>
<li><strong>Event-based</strong>: Invalidate on specific events</li>
<li><strong>Manual</strong>: Explicit invalidation via API</li>
<li><strong>Dependency-based</strong>: Track data dependencies</li>
</ol>
<h2>Practical Implementation</h2>
<pre><code class="language-go">type CacheEntry struct {
    Value      interface{}
    ExpiresAt  time.Time
    AccessedAt time.Time
    Frequency  int64
}

type Cache struct {
    data map[string]*CacheEntry
    mu   sync.RWMutex
}

func (c *Cache) Get(key string) (interface{}, bool) {
    c.mu.RLock()
    defer c.mu.RUnlock()
    
    entry, ok := c.data[key]
    if !ok || time.Now().After(entry.ExpiresAt) {
        return nil, false
    }
    
    entry.AccessedAt = time.Now()
    entry.Frequency++
    return entry.Value, true
}
</code></pre>
<h2>Redis in Production</h2>
<p>Redis is the go-to choice for caching at scale:</p>
<ul>
<li><strong>In-memory</strong>: Extremely fast</li>
<li><strong>Persistence</strong>: Optional RDB or AOF</li>
<li><strong>Replication</strong>: Master-slave setup</li>
<li><strong>Clustering</strong>: Horizontal scaling</li>
</ul>
<h2>Monitoring and Metrics</h2>
<p>Track these metrics to optimize your cache:</p>
<ul>
<li><strong>Hit Rate</strong>: Percentage of cache hits</li>
<li><strong>Eviction Rate</strong>: Items removed due to capacity</li>
<li><strong>Memory Usage</strong>: Cache size and efficiency</li>
<li><strong>Response Time</strong>: Impact on overall latency</li>
</ul>
<h2>Conclusion</h2>
<p>Effective caching requires understanding your access patterns, choosing the right strategy, and careful monitoring. Start simple, measure, and optimize based on real-world performance data.</p>

<h2>Kubernetes Networking Fundamentals</h2>
<p>Kubernetes provides a sophisticated networking model that enables secure, scalable communication between containers. Understanding this model is essential for running production-grade applications.</p>
<h2>Pod Networking</h2>
<p>Every pod gets a unique IP address on a flat network where all pods can communicate directly.</p>
<h3>Pod-to-Pod Communication</h3>
<p>Pods communicate directly via IP addresses:</p>
<pre><code>Pod A (10.0.0.1) &lt;-&gt; CNI Bridge &lt;-&gt; Pod B (10.0.0.2)
</code></pre>
<p>The Container Network Interface (CNI) manages this connectivity.</p>
<h3>Network Plugins</h3>
<p>Popular CNI implementations:</p>
<ul>
<li><strong>Flannel</strong>: Simple overlay networks</li>
<li><strong>Calico</strong>: BGP-based with network policies</li>
<li><strong>Weave</strong>: Multi-host mesh networking</li>
<li><strong>Cilium</strong>: eBPF-based networking</li>
</ul>
<h2>Services: Exposing Pods</h2>
<p>Services provide stable endpoints for accessing pods:</p>
<pre><code class="language-yaml">apiVersion: v1
kind: Service
metadata:
  name: web-service
spec:
  type: ClusterIP
  selector:
    app: web
  ports:
  - port: 80
    targetPort: 8080
</code></pre>
<h3>Service Types</h3>
<p><strong>ClusterIP</strong>: Internal service, only accessible within the cluster</p>
<p><strong>NodePort</strong>: Exposes service on a node port (30000-32767)</p>
<p><strong>LoadBalancer</strong>: Cloud provider&#39;s load balancer</p>
<p><strong>ExternalName</strong>: DNS CNAME to external service</p>
<h2>Ingress Controllers</h2>
<p>Ingress manage external access to services:</p>
<pre><code class="language-yaml">apiVersion: networking.k8s.io/v1

metadata:
  name: web-ingress
spec:
  rules:
  - host: example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: web-service
            port:
              number: 80
</code></pre>
<h3>Popular Ingress Controllers</h3>
<ul>
<li><strong>NGINX Ingress</strong>: Most widely used</li>
<li><strong>Traefik</strong>: Cloud-native and dynamic</li>
<li><strong>HAProxy Ingress</strong>: High performance</li>
<li><strong>AWS ALB Ingress</strong>: AWS-native</li>
</ul>
<h2>Network Policies</h2>
<p>Control traffic between pods:</p>
<pre><code class="language-yaml">apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-external
spec:
  podSelector:
    matchLabels:
      tier: frontend
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          tier: backend
    ports:
    - protocol: TCP
      port: 8080
</code></pre>
<h2>DNS in Kubernetes</h2>
<p>Kubernetes includes built-in DNS:</p>
<pre><code>&lt;service-name&gt;.&lt;namespace&gt;.svc.cluster.local
</code></pre>
<h2>Performance Optimization</h2>
<h3>Connection Pooling</h3>
<p>Reuse connections to reduce overhead:</p>

<p>Istio and Linkerd provide advanced networking:</p>
<ul>

<li>Observability</li>

<h2>Understanding Query Plans</h2>
<p>Use EXPLAIN ANALYZE to understand how PostgreSQL executes queries:</p>
<pre><code class="language-sql">EXPLAIN ANALYZE
SELECT users.name, COUNT(orders.id) as order_count
FROM users
LEFT JOIN orders ON users.id = orders.user_id
GROUP BY users.id;
</code></pre>
<p>Look for:</p>
<ul>
<li>Sequential scans (usually bad)</li>
<li>Index scans (usually good)</li>
<li>High actual vs. estimated rows (statistics issue)</li>
</ul>
<h2>Indexing Strategies</h2>
<h3>B-Tree Indexes</h3>
<p>The default index type for most scenarios:</p>

21:T852,<h1>Implementing Circuit Breaker Pattern</h1>
<h2>Overview</h2>
<p>Circuit breaker prevents repeated calls to failing services:</p>
<pre><code>CLOSED → checks requests → OPEN (on failure) → HALF_OPEN (after timeout) → CLOSED
</code></pre>
<h2>States</h2>
<h3>CLOSED (Normal)</h3>
<p>Requests pass through normally. Failures counted.</p>
<h3>OPEN (Failing)</h3>
<p>Circuit breaks, fast-fail without calling service.</p>
<pre><code class="language-go">if circuitOpen {
    return errors.New(&quot;circuit breaker open&quot;)
}
</code></pre>
<h3>HALF_OPEN (Recovery)</h3>
<p>Limited requests allowed to test if service recovered:</p>
<pre><code class="language-go">if circuitHalfOpen &amp;&amp; testCount &lt; maxTestRequests {
    // Allow request
    return callService()
}
</code></pre>
<h2>Implementation</h2>
<pre><code class="language-go">type CircuitBreaker struct {
    state       string
    failures    int
    maxFailures int
    lastFailTime time.Time
    timeout     time.Duration
}

func (cb *CircuitBreaker) Call(fn func() error) error {
    if cb.state == &quot;OPEN&quot; {
        if time.Since(cb.lastFailTime) &gt; cb.timeout {
            cb.state = &quot;HALF_OPEN&quot;
            cb.failures = 0
        } else {
            return ErrCircuitOpen
        }
    }
    
    err := fn()
    if err != nil {
        cb.failures++
        cb.lastFailTime = time.Now()
        if cb.failures &gt;= cb.maxFailures {
            cb.state = &quot;OPEN&quot;
        }
    } else {
        cb.failures = 0
        cb.state = &quot;CLOSED&quot;
    }
    return err
}
</code></pre>
<h2>Libraries</h2>
<ul>
<li><strong>Hystrix</strong>: Netflix&#39;s implementation</li>
<li><strong>Gobreaker</strong>: Go implementation</li>
<li><strong>Resilience4j</strong>: Java implementation</li>
</ul>
<h2>Fallback Strategies</h2>
<pre><code class="language-go">result, err := circuitBreaker.Call(func() error {
    return fetchData()
})
if err != nil &amp;&amp; isFallbackAvailable {
    return getFallbackData()
}
</code></pre>

22:T761,<h1>Building Resilient APIs with Retry Logic</h1>
<h2>Why Retries Matter</h2>
<p>Transient failures (network hiccups, timeouts) should not fail requests:</p>
<pre><code>Request 1 → Timeout
Request 2 (after 100ms) → Timeout  
Request 3 (after 200ms) → Success ✓
</code></pre>
<h2>Retry Strategy</h2>
<h3>Exponential Backoff with Jitter</h3>
<pre><code class="language-go">func exponentialBackoff(attempt int) time.Duration {
    baseDelay := 100 * time.Millisecond
    maxDelay := 30 * time.Second
    
    delay := baseDelay * time.Duration(math.Pow(2, float64(attempt)))
    if delay &gt; maxDelay {
        delay = maxDelay
    }
    
    // Add jitter: ±10%
    jitter := time.Duration(rand.Intn(20)-10) * delay / 100
    return delay + jitter
}
</code></pre>
<h2>Implementation</h2>
<pre><code class="language-go">func retryWithBackoff(fn func() error, maxAttempts int) error {
    for attempt := 1; attempt &lt;= maxAttempts; attempt++ {
        err := fn()
        if err == nil {
            return nil
        }
        
        if attempt &lt; maxAttempts {
            delay := exponentialBackoff(attempt - 1)
            time.Sleep(delay)
        }
    }
    return ErrMaxRetriesExceeded
}
</code></pre>
<h2>Idempotency</h2>
<p>Retries only safe for idempotent operations:</p>
<pre><code class="language-go">// Safe (GET)
GET /users/123

// Unsafe (POST without idempotency)
POST /orders {&quot;item&quot;: &quot;book&quot;}

// Safe (POST with idempotency key)
POST /orders {&quot;item&quot;: &quot;book&quot;} 
Headers: {&quot;Idempotency-Key&quot;: &quot;unique-uuid&quot;}
</code></pre>
<h2>When NOT to Retry</h2>
<ul>
<li>4xx errors (client errors)</li>
<li>Permanent failures</li>
<li>Non-idempotent operations</li>
<li>Slow operations (already waited)</li>
</ul>
<p>Conclusion: Smart retry logic significantly improves system resilience and user experience.</p>
